Step 02 of 06

Write the Right Prompt

A great Claude prompt is a creative brief. Six parts. Each one removes ambiguity and pushes output quality up a level.

Part 01

Role

Tell Claude who it is. Be specific — "senior UI/UX designer and front-end developer specialising in B2B SaaS prototypes."

Part 02

Context

What's being built, who the audience is, when it will be shown, and what decision it needs to drive.

Part 03

Brand Rules

Hex codes, font names, tone-of-voice. Don't let Claude invent your brand identity from scratch.

Part 04

UX Intent

What interactions should exist? Tabs? Hover states? A multi-step form? Scroll-reveal animations? Be explicit.

Part 05

Constraints

Single HTML file. Inline CSS only. No external dependencies. Mobile-responsive. Code commented section-by-section.

Part 06

Output Format

HTML + inline CSS, Webflow-ready classes, or a React component. Specify exactly what Claude hands back.

Step 03 of 06

Generate Usable HTML

Ask Claude to produce a single, self-contained HTML file. The golden rule: everything in one file, everything readable, everything tweakable — even by a non-developer.

  • Inline CSS only — no external stylesheets. One file = zero dependencies = instantly shareable over email, Notion, Slack.
  • Responsive layout — specify breakpoints, or say "mobile-first, responsive." Claude knows what this means.
  • Dummy content that feels real — tell Claude to use realistic placeholder content, not Lorem ipsum. Realism sells the concept.
  • Comments in the code — ask Claude to annotate each section. Your devs will thank you during feasibility review.
  • No frameworks or build steps — pure HTML + CSS + vanilla JS. Opens in any browser, sends over email, embeds in Notion.

Step 04 of 06

Test & Iterate

The first output is a 70% draft. That's the point — speed to tangible beats perfection in planning. Now refine it in conversation with Claude.

4a
Refine Interactions
Don't rewrite the full prompt. Reply to Claude: "Make the button hover more dramatic. Add a transition on the tab switch. Remove the sidebar." Iterate in-thread.
4b
Improve Visual Fidelity
Once the logic works, push the finish. "Make it feel more premium. Tighten spacing. Use a more sophisticated font hierarchy." Watch it sharpen fast.
4c
Reduce Complexity
Ask Claude: "What's the most important interaction? Remove everything else." Scope clarity through constraint forces real prioritisation — fast.

Step 05 of 06

Dev Feasibility Check

Once you have something your team reacts to, take it to engineering — not as a spec, but as a question. The demo earns the conversation.

  • Embed test — drop the HTML into your staging environment or Notion. Does it load cleanly? Any conflicts with existing styles?
  • Performance impact — run Lighthouse on the demo. Flag animation-heavy sections that may need re-implementation in production.
  • Production constraints — which interactions are straightforward in your stack? Which require a library or custom component?
  • Scope signal — the demo lets devs estimate time on real, visible things rather than abstract tickets. "The wizard: 3 days. Drag-drop: 2 weeks."

Instead of building monolithic features: Sketch → Claude demo → Dev feasibility → Iterate.

Faster feedback. Clearer scope. Reusable assets. It's a brilliant loop.
